## 详细发现与修复

### D：数据层缺陷（两个 P0）

1. **clip 字段映射错误**（persist_script_results）
   - 历史 v0.9.0 的代码传 `title`/`sort_order`/`project_id` 给 `Production.create_clip`
   - Clip schema cast 列表：`episode_id / clip_index / start_time / end_time / duration / content / summary / location / characters / props / screenplay` — 那 3 个字段都不在列表里
   - Ecto 静默丢弃 → 数据全丢
   - 修复：重写 persist_script_results，传对的字段名，并把 LLM clip-level characters 用新增的 `sync_global_assets/4` 聚合到全局表

2. **LLM array vs schema string 静默失败**（ClipsBuild handler）
   - LLM 返回 `characters: ["林岳", "神秘少女"]` 是 JSON 数组
   - Clip.characters 是 `:string`
   - 直接 cast 导致**校验失败但不抛错**（Changeset 记录 error 但 insert 还是继续或返回 error tuple 被忽略）
   - 之前的 `to_string_field/1` 在另一个模块定义，ClipsBuild 用不到
   - 修复：在 ClipsBuild 里本地定义 `to_string_field` + 应用到所有相关字段

3. **script_to_storyboard watchdog 误判**
   - 真实数据：13 panel 顺序 LLM 调用 ≈ 6 分钟
   - 原 watchdog 5 分钟阈值 → 任务实际跑完但被标记为 `failed watchdog_timeout`
   - 修复：`@stale_threshold_seconds = 600`（10 分钟），并在 TaskScheduler 里加注释说明

## 技术决策记录

- **不用 Ecto.Multi 在 sync_global_assets 里原子性插入**：因为这是 best-effort 同步，失败不应回滚已创建的 clips
- **watchdog 阈值提到 10 分钟而不是让 handler 主动 touch_heartbeat**：后者更优雅但需要改多个 handler；前者一行改动覆盖所有慢任务。如果再有更慢任务出现再考虑 heartbeat 方案
- **长文本不做强制分集弹窗**：原项目是 modal，我用内联 alert banner 减少打断。用户仍可选择右侧集数下拉或直接提交
- **B4 提示条只是 hint 不是 blocker**：用户可以直接点"开始创作"按单集处理，系统后续管线会自己拆 clips

## 4 个连环 P0 全链路排雷

从"切换配置 + 改 prompt" 开始，连续踩了 4 个独立 bug：

### P0-1: 模型名双重 transform
- 现象：apiyi 返回 "当前分组下对于模型 **veo-3.1-fast-landscape-fl** 计费模式无可用渠道"
- 模型名**顺序错了**（应该 `veo-3.1-landscape-fast-fl`，landscape 在 fast 前）
- 根因：VideoPanel handler 调 `apply_model_suffix` 把后缀拼到末尾 → "veo-3.1-fast-landscape-fl"；然后 apiyi provider `transform_veo_model/1` 看到已含 -landscape 不改
- 修：handler 不做任何 transform，完全让 apiyi 内部处理（v0.9.6 的想法反了，我之前以为"只对 apiyi 应用" apply_model_suffix，实际应该"完全跳过"）

### P0-2: Req multipart 格式
- 现象：`%Protocol.UndefinedError{protocol: Enumerable, value: {"input-reference.png", <<bytes>>, ...}}`
- 根因：build_file_part 返回 `{field, filename, data, opts}` 4-tuple，但 Req 期望 `{field, {body, opts}}` 2-tuple
- 修：改签名 `{field_name, {data, filename: default_filename, content_type: content_type}}`

### P0-3: AsyncPoller config 查找
- 现象：`%KeyError{key: :api_key, term: %{}}` 每 15 秒刷一次，直到 watchdog 10 分钟超时
- 根因：external_id 前缀 "OPENAI"（apiyi 的历史遗留）→ `Map.get(user_config, "OPENAI", %{})` → 找不到，返回空 map → 调 poll_task 时 Map.fetch! api_key 崩
- 修：`config_key = provider_key_from_prefix(provider)` 先映射 "OPENAI" → "apiyi"，再 Map.get

### P0-4: external_id 4 段解析
- 现象：apiyi 返回 `"task_not_exist"`
- 根因：external_id 是 `OPENAI:VIDEO:YXBpeWk:video_7746840a-...`（4 段，apiyi 额外加了 base64 token "YXBpeWk" = "apiyi"）
- parse_external_id 只 split parts=3 → rest = `"YXBpeWk:video_7746840a-..."`
- 当做 video_id 传给 apiyi poll API → 404
- 修：do_poll 里再 split 一次，`Base.url_decode64` 验证前段是 token 就剥离，留下真正的 `video_7746840a-...`

### 附带 P1: header list format
- Req 新版对 `{"content-type", ["application/json; charset=utf-8"]}` 返回值是 list
- `String.split(ct, ";")` 在 ct 是 list 时崩
- 加 `{"content-type", [ct | _]} when is_binary(ct)` 分支

## 功能增强

**auto-FL (first-last frame) 模式**：
```elixir
defp find_next_panel_image(panel) do
  Production.list_panels(panel.storyboard_id)
  |> Enum.sort_by(& &1.panel_index)
  |> Enum.drop_while(&(&1.id != panel.id))
  |> Enum.drop(1)
  |> Enum.find_value(&(is_binary(&1.image_url) and &1.image_url != "" && &1.image_url))
end
```

每个 panel 的视频生成默认启用 FL 模式，用**下一个 panel 的 image** 作为 last_frame，使连续镜头过渡自然。

**VEO 3.1 模型自动选择**：
- aspect_ratio 16:9 + 有 first_frame_image + veo-3.1-fast base model
- apiyi provider `transform_veo_model` 自动生成 `veo-3.1-landscape-fast-fl`
- use_frame_mode = true → walk multipart 上传双 image

## 内容审核观察

VEO 3.1 触发 "图片中包含未成年人，请重新上传" **3/3 批量失败**：
- Panel 93279843 "少女" 特写镜头 → failed
- Panel d7be4d82 "少女苏醒" → failed
- Panel b7d96e1b "少女冲向" → failed

但 Panel 06347c45 "Lin Yue crouching" 成功。结论：
- VEO 的视觉审核模型对"年轻女性 + 特写"组合较敏感
- prompt 里 "young woman" 本身不会触发，但图像里人物外观被判年轻就失败
- 绕过方式：
  1. Prompt 显式写 "adult woman in her late 20s"
  2. 图像生成时约束年龄段
  3. 或换 provider（MiniMax Hailuo 对未成年审核较宽松）

这是 **provider 层策略问题**，不是代码 bug。
